# Phase 2B v9 — Compliance Baseline Audit

Audit date: 2026-05-14. Verified from code only, not docs/plans.
SMS vendor migrated Twilio → Telnyx (2026-05-11). Twilio symbols stripped from runtime; only webhook URL comments + CLAUDE.md still reference Twilio. Audit reads "Twilio path" as "Telnyx path."

## Summary

Compliance core (STOP suppression, quiet hours, Stripe idempotency, duplicate-payment prevention) is real and tested. Three soft spots:

1. **TCPA opt-in not enforced on the outbound send gate.** `sms_compliance.send_sms` blocks opt-outs and quiet hours but does NOT call `has_opted_in()`. Only the Cora agent path enforces opt-in (via `compliance_check` gating tool). Any non-agent caller (e.g. `send_opt_in_prompt`, sample-leads SMS, sms_commands replies) can SMS a number with zero recorded consent.
2. **DNC helper exists but is dead code.** `sms_compliance.check_dnc` is implemented and untested — no caller invokes it.
3. **Dead-letter queue is Postgres, not Redis.** `SmsDeadLetter` table is the DLQ. CLAUDE.md / sms_compliance docstring promised Redis in "2B-2"; not done. STOPs route to DLQ (`reason="opt_out"`) via PG insert — functional but not the Redis design.

Also: `sms_commands._find_subscriber` is a stub that always returns `None` (waiting on `Subscriber.phone` column). Every inbound product command (LOCK/BALANCE/TOPUP/...) currently replies "Reply HELP to get started" regardless of sender. A2P/10DLC is an external dependency — no code, only `telnyx_messaging_profile_id` env field + `brand_id`/`campaign_id: null` placeholders in finetuner configs.

## Pending Work Only

### Must fix before launch
1. **`sms_dead_letters` CHECK constraint vs `quiet_hours` reason** — `sms_compliance.send_sms:211` writes `reason="quiet_hours"`, but `models.py:1734` and the alembic migration restrict reason to `{opt_out, delivery_failed, error, unresolvable}`. Quiet-hours suppressions raise on flush and the DLQ row is lost. Either widen the CHECK or map to `error` with a metadata flag. (Code-only fix.)
2. **Wire `handle_opt_in_reply` into the Telnyx inbound webhook** — `src/api/main.py:2556` currently lets YES fall through to `sms_commands.parse` (which treats YES as pause-confirm). New TCPA opt-ins via "YES" from numbers without a `pause_pending` Redis key are silently dropped.
3. **Enforce `has_opted_in()` inside `sms_compliance.send_sms` for marketing sends** — today only the agent tool `compliance_check` enforces opt-in. Any other caller (sample-leads SMS, missed-call welcome, sms_commands reply) can SMS a non-consented number. Plumb a `message_type` arg or check `task_type` against a marketing-classification list.
4. **Stub `sms_commands._find_subscriber`** — `services/sms_commands.py:65` returns None for every inbound, so every product command replies "Reply HELP to get started." Add `Subscriber.phone` column (referenced in the TODO comment) and resolve from there.

### Should fix soon
5. **Either wire `check_dnc` into the send path or delete it.** Dead helper is a compliance footgun — implies coverage that doesn't exist.
6. **Persist a `MessageOutcome` row for every `sms_compliance.send_sms` call** (sent or suppressed) so non-agent paths have the same auditability as Cora graph sends. Today suppression audit lives only in `SmsDeadLetter`; delivery audit lives only in agent `write_tools.send_sms`.
7. **Tighten phone normalization** — `_normalize` is whitespace-only. Mixed `+1` / `1` / 10-digit storage will split keys and miss opt-outs.
8. **Resolve CLAUDE.md drift** — docs still say Twilio for SMS; runtime is Telnyx. Stripe webhook docstring still says "before every Twilio send" in `sms_compliance.can_send:93`.
